  <name>Task 1: Fix double-denormalization in M3GNet and TensorNet training evaluation</name>
  <files>cathode_ml/models/train_m3gnet.py, cathode_ml/models/train_tensornet.py, tests/test_m3gnet.py, tests/test_tensornet.py</files>
  <behavior>
    - Test: When _run_lightning_training returns data_mean=2.0, data_std=3.0 and predict_with_m3gnet returns [1.5, 2.3, 0.8], the predictions passed to compute_metrics should be [1.5, 2.3, 0.8] (NOT [1.5*3+2=6.5, 2.3*3+2=8.9, 0.8*3+2=4.4])
    - Test: Same behavior for TensorNet -- predict_with_tensornet output used directly without rescaling
    - Test: compute_metrics receives y_pred values matching predict_with_X output (no transformation applied)
  </behavior>
  <action>
**The bug:** In `train_m3gnet_for_property()` (lines 262-264) and `train_tensornet_for_property()` (lines 264-266), after calling `predict_with_X(model, test_structures)`, the code applies `predictions = [p * data_std + data_mean for p in raw_predictions]`. This is wrong because matgl's `predict_structure()` (called inside `predict_with_X`) already returns values in the original (denormalized) scale. The `ModelLightningModule` handles normalization internally during training and denormalizes on prediction.

**The fix in train_m3gnet.py `train_m3gnet_for_property()`:**
1. Remove the `data_mean, data_std = _run_lightning_training(...)` return value capture (or ignore it).
2. Replace lines 263-264:
   ```python
   raw_predictions = predict_with_m3gnet(model, test_structures)
   predictions = [p * data_std + data_mean for p in raw_predictions]
   ```
   With:
   ```python
   # predict_structure() already returns denormalized values --
   # ModelLightningModule handles normalization internally
   predictions = predict_with_m3gnet(model, test_structures)
   ```
3. Change `y_pred = np.array(predictions)` to use the direct predictions.

**The fix in train_tensornet.py `train_tensornet_for_property()`:**
1. Same pattern -- remove the manual denormalization at lines 265-266.
2. Replace:
   ```python
   raw_predictions = predict_with_tensornet(model, test_structures)
   predictions = [p * data_std + data_mean for p in raw_predictions]
   ```
   With:
   ```python
   # predict_structure() already returns denormalized values --
   # ModelLightningModule handles normalization internally
   predictions = predict_with_tensornet(model, test_structures)
   ```

**Update the existing test mocks** in both test files: The tests that patch `predict_with_m3gnet`/`predict_with_tensornet` and `compute_metrics` should verify that `compute_metrics` receives the raw prediction values, not rescaled ones.

**Add explicit denormalization tests:**
- In `tests/test_m3gnet.py`, add a test class `TestNoDenormalization` with a test that verifies `train_m3gnet_for_property` passes predict_with_m3gnet output directly to compute_metrics without transformation. Mock `_run_lightning_training` to return known data_mean/data_std, mock `predict_with_m3gnet` to return known values, and use a spy on `compute_metrics` to capture the y_pred argument. Assert y_pred matches the mock return, NOT the rescaled version.
- Same pattern in `tests/test_tensornet.py`.

Note: `_run_lightning_training` return type signature can keep returning `data_mean, data_std` (it does not need to change), but the caller should not use those values for post-hoc denormalization. The return values may still be useful for logging or saving to checkpoint metadata.

<task type="auto">
  <name>Task 2: Fix TensorNet training log duplication</name>
  <files>cathode_ml/models/train_tensornet.py, cathode_ml/models/train_m3gnet.py</files>
  <action>
**The bug:** TensorNet (and potentially M3GNet) training produces duplicated log lines because Lightning's `enable_progress_bar=True` combined with Python's logging module and Lightning's internal logging creates redundant output. The CSVLogger's `metrics.csv` has separate rows for train and val metrics per epoch (by design), but the console output gets duplicated.

**The fix in train_tensornet.py `_run_lightning_training()`:**
1. Change `enable_progress_bar=True` to `enable_progress_bar=False` on the `L.Trainer()` constructor (line ~182). The project uses Python logging for status updates, not the progress bar.
2. Add `enable_model_summary=False` to suppress the redundant model summary that Lightning prints.
3. Ensure only one logging handler is active by checking for existing Lightning log handlers. Before the `trainer.fit()` call, suppress Lightning's root logger duplication:
   ```python
   # Prevent Lightning from adding duplicate console handlers
   logging.getLogger("lightning.pytorch").setLevel(logging.WARNING)
   ```

**Apply the same fix in train_m3gnet.py `_run_lightning_training()`:**
1. Same three changes: `enable_progress_bar=False`, `enable_model_summary=False`, and Lightning logger level set to WARNING.

This ensures training output is clean: one set of metrics per epoch via the CSVLogger, with the project's own Python logger handling status messages.
